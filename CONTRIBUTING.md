# Правила работы в команде

## 🔀 Git Flow

### Ветки
- `main` — только готовый, рабочий код. Прямые коммиты **запрещены**.
- `dev` — основная ветка разработки. Merge через PR.
- `feature/*` — ваша личная ветка для разработки.

### Как работать

```bash
# 1. Синхронизируйся с dev перед началом работы
git checkout dev
git pull origin dev

# 2. Создай или переключись на свою ветку
git checkout feature/your-module

# 3. Пиши код, коммить часто с понятными сообщениями
git add .
git commit -m "feat: добавить ClientHandler с базовой обработкой пакетов"

# 4. Пушь в свою ветку
git push origin feature/your-module

# 5. Создай Pull Request в dev через GitHub
```

## 📝 Формат коммитов

```
<тип>: <краткое описание на русском или английском>

Типы:
  feat:     новая функциональность
  fix:      исправление бага
  refactor: рефакторинг без изменения поведения
  docs:     документация
  style:    форматирование, отступы
  test:     тесты
```

**Примеры:**
```
feat: добавить UserDAO с методами findByEmail и save
fix: исправить NPE в ClientHandler при разрыве соединения
refactor: вынести логику аутентификации в AuthService
docs: обновить README — инструкция по запуску сервера
```

## ⚠️ Важные правила

1. **Никогда не пушь `db.properties`** — там пароли. Файл в `.gitignore`.
2. **Shared-классы** (`model/`, `protocol/`) — изменяй только после согласования с командой, т.к. они влияют на всех.
3. **Не ломай `dev`** — перед merge убедись, что код компилируется.
4. **Конфликты** — решай вместе с тем, чей код конфликтует.

## 📋 Checklist перед Pull Request

- [ ] Код компилируется (`mvn clean compile`)
- [ ] Нет хардкода (паролей, путей, IP)
- [ ] Добавлены комментарии к публичным методам
- [ ] `db.properties` НЕ добавлен в коммит
